# 多周期架构重构设计规格

## 背景与问题

### 现象
在加载 500 根 K 线时，笔和段的识别结果基本正确；加载 1000 根 K 线时，结果明显偏离预期——段过多、结构混乱、中枢错位。

### 根本原因
当前架构用**数据量**模拟**级别**。缠论的级别由时间周期决定，不是由同一周期内看多少根 K 线决定的。

具体表现：
- `min_kline_gap`、`minSegmentRatio` 是全局固定参数，对 500 根和 1000 根 K 线使用相同阈值，导致在数据量大时小波动与主结构地位等同
- 段检测的贪心算法在更长序列上过度细分，把震荡内部切割成多个段
- `VALLEY_EXTEND_TOLERANCE` 基于 `max(tp) - min(tp)` 计算，K 线越多数据范围越大，相对阈值失真
- 代码没有"级别"的概念，只有"笔级别"和通过递归模拟的"段级别"，两者混用同一参数空间

### 正确的缠论级别对应关系

```
1分钟  K线 → 1分钟级别的笔和段  →  上一级别参考：5分钟
5分钟  K线 → 5分钟级别的笔和段  →  上一级别参考：30分钟
15分钟 K线 → 15分钟级别的笔和段 →  上一级别参考：1小时
30分钟 K线 → 30分钟级别的笔和段 →  上一级别参考：4小时
4小时  K线 → 4小时级别的笔和段  →  上一级别参考：日线
日线        → 日线级别的笔和段   →  上一级别参考：周线
周线        → 周线级别的笔和段   →  上一级别参考：月线
```

同一时间段内，上下文周期图的"一笔"对应主周期图的"一段"甚至"多段+中枢"。这才是真正的级别关系，不能用"取多少根 K 线"来替代。

**常用操作周期建议**：实盘中最实用的三个操作周期是 5m、15m、30m，分别对应 30m、1h、4h 作为上下文参照，在这三个档位上级别关系紧凑、结构对应清晰，是低级别找入场点、高级别判断方向的标准用法。

## 解决方案总览

引入**双周期分析模型**：

- **主周期（Primary）**：用户当前选择的分析周期，画笔和段，检测中枢和买卖点
- **上下文周期（Context）**：自动配对的更高周期，画更高级别的段和中枢，提供结构参照

两套数据独立获取、独立计算、分层渲染在同一图上。

同时，**每个周期固定最大 K 线数量**（默认 300 根），彻底消除"数据量影响结构判断"的问题。用户想看更长历史，切换到更大周期，而不是加载更多 K 线。

## 第一层：数据层重构

### 1.1 周期配对规则

每个主周期自动配对一个上下文周期：

| 主周期 | 上下文周期 | 倍数 | 备注 |
|--------|-----------|------|------|
| 1m     | 5m        | ×5   | |
| 3m     | 15m       | ×5   | |
| 5m     | 30m       | ×6   | 常用操作周期 |
| 15m    | 1h        | ×4   | 常用操作周期 |
| 30m    | 4h        | ×8   | 常用操作周期 |
| 1h     | 4h        | ×4   | |
| 2h     | 1d        | ×12  | |
| 4h     | 1d        | ×6   | |
| 6h     | 1d        | ×4   | |
| 1d     | 1w        | ×7   | |
| 1w     | 1M        | ×4   | |
| 1M     | —         | —    | 无上下文 |

**配对原则**：上下文周期是主周期的 4–8 倍，保证主周期的"一段"在上下文周期中大约对应"一笔到几笔"的长度，两个级别之间结构上有直接的对应关系，可以互相印证。

**关于 `30m → 4h`（×8）**：倍数稍大于其他档位，但 4h 是市场上机构和主流交易者普遍参考的周期，信号质量高，且"上下文周期变化慢"本身就是价值所在——它告诉你更大的方向，不需要频繁更新。

**用户可覆盖**：配对关系作为默认值内置，但允许用户在 UI 上手动选择上下文周期，因为不同交易者对级别的理解有个人习惯。

### 1.2 数据量规范

| 周期       | 最大 K 线数 | 覆盖时长参考 |
|------------|------------|------------|
| 1m–3m      | 300 根     | 5–15 小时  |
| 5m–15m     | 300 根     | 1–3 天     |
| 30m–1h     | 300 根     | 6–12.5 天  |
| 2h–4h      | 300 根     | 25–50 天   |
| 6h–1d      | 300 根     | 75–300 天  |
| 1w–1M      | 200 根     | 4–16 年    |

固定数据量是解决"数据量影响结构判断"问题的根本手段。每个周期只在自己的"合理时间窗口"内分析结构，不跨越窗口。

前端不再暴露"K 线数量"的输入框，改为用户只选择周期。如需回溯历史，通过日期选择器指定起止时间，而不是增加 K 线数量。

### 1.3 数据获取流程变更

**现有流程：**
```
用户输入 symbol + interval + limit
  → fetch_klines(symbol, interval, limit)
  → 单一 K 线序列
  → 单次全量计算
```

**新流程：**
```
用户输入 symbol + primary_interval
  → 自动推导 context_interval
  → 并行 fetch:
      fetch_klines(symbol, primary_interval, PRIMARY_LIMIT)
      fetch_klines(symbol, context_interval, CONTEXT_LIMIT)
  → 两套 K 线序列独立存储
  → 两套独立管道并行计算
```

两个 fetch 请求可以并发发出，不需要串行等待。

### 1.4 时间对齐

主周期和上下文周期的 K 线在时间轴上需要对齐渲染。每根上下文周期 K 线的 `openTime` 落在主周期 K 线时间轴的某个位置，渲染时用这个位置作为锚点。

对齐逻辑：对每个上下文分型/笔端点，找主周期 K 线序列中 `openTime` 最接近的那根，用其 x 坐标渲染。

## 第二层：算法层标准化

### 2.1 废除动态 `minSegmentRatio`

`minSegmentRatio` 存在的原因是同一序列里波动幅度差异巨大。固定数据量后，每个周期的波动幅度分布趋于稳定，不再需要动态调整。

**变更：**
- 后端 `compute_auto_segments` 的 `min_segment_ratio` 参数保留，但固定为 `0.0`（不过滤），让缠论规则本身决定段的边界
- 前端移除 `minSegmentRatio` 滑块（或降级为调试用的隐藏选项）
- PoP 回退算法的 `_merge_small_segments` 中的 `min_range` 同样归零，让 PoP 回退只依赖结构规则

### 2.2 修复线段判定条件

当前 `_higher_high_higher_low` 只检查谷值（低点），未检查高点。`_lower_high_lower_low` 同理。

正确的缠论线段条件是"更高的高点 **且** 更高的低点"两者同时满足，需要补充高点验证逻辑。这个修复与多周期改造解耦，可以独立完成，但建议在同一批次处理。

### 2.3 参数归一化

每个周期的算法实例使用独立的固定参数集：

```
PRIMARY 参数集：
  min_kline_gap = 4       （5根K线间隔，缠论标准）
  min_segment_ratio = 0.0
  max_segment_strokes = 15

CONTEXT 参数集：
  min_kline_gap = 4       （上下文周期自己的K线级别，同样标准）
  min_segment_ratio = 0.0
  max_segment_strokes = 15
```

两套参数集相同，因为每套数据在自己的周期坐标系内都是"标准"的 K 线，不需要额外调整。

### 4.1 状态结构变更

**现有：**
```javascript
const S = {
```

**新结构：**
```javascript
const S = {
  // 主周期数据（原来的数据）
  primary: {
    klines: null,
    fractals: [],
    turningPoints: [],
    strokes: [],
    segments: [],
    zhongshu: [],
    higherSegments: [],
```

### 4.2 状态访问规范

所有原来直接访问 `S.turningPoints`、`S.segments` 的地方改为 `S.primary.turningPoints`、`S.primary.segments`。

### 4.3 resetUIState 调整

`resetUIState()` 分为两个独立函数：

- `resetPrimary()`：清空主周期数据和相关 UI 状态
- `resetAll()`：清空所有数据（切换交易对时调用）

切换参数重算时只调用 `resetPrimary()`，上下文周期数据保持不动（除非切换了 symbol 或 primaryInterval 导致 contextInterval 变化）。

## 第六层：数据持久化设计

### 6.1 持久化策略总览

不同类型的数据，持久化价值和成本差异很大，策略分三类：

| 数据类型 | 是否落盘 | 原因 |
|---------|---------|------|
| K线数据 | **是**，SQLite 增量缓存 | 来自外部网络，拉取有限制，历史数据不变 |
| 自动计算的笔/段/中枢 | **否**，每次实时重算 | 计算成本极低（毫秒级），末尾数据随新K线回溯，缓存失效逻辑复杂 |
| 用户手动标注 | **是**，JSON 文件 | 用户主观判断，无法通过算法还原，必须持久化 |

核心原则：**省网络IO用SQLite缓存K线，省用户劳动用JSON保存标注，算法输出不缓存。**

---

### 6.2 K线数据落盘

#### 存储方案：SQLite

单文件数据库，零运维，Python 内置支持，足以应对个人工具的数据量。

**数据量估算**（单条约 60 字节）：

| 周期 | 1年数据量 | 大小 |
|------|---------|------|
| 5m   | ~105,000 根 | ~6MB |
| 15m  | ~35,000 根  | ~2MB |
| 30m  | ~17,500 根  | ~1MB |
| 1h   | ~8,760 根   | ~0.5MB |
| 4h   | ~2,190 根   | ~0.1MB |

10 个交易对、全周期存储 1 年，总量不超过 **200MB**，完全不是问题。

#### 表结构

```
表名：klines
字段：
  symbol      TEXT     交易对，如 BTCUSDT
  interval    TEXT     周期，如 5m、1h
  open_time   INTEGER  开盘时间戳（毫秒），唯一标识一根K线
  open        REAL
  high        REAL
  low         REAL
  close       REAL
  volume      REAL
  close_time  INTEGER  收盘时间戳（毫秒）

主键：(symbol, interval, open_time)
索引：(symbol, interval, open_time DESC)  ← 覆盖最常见的"取最近N根"查询
```

不同周期、不同交易对共用同一张表，通过 `symbol + interval` 区分，查询简单，管理方便。

#### 增量更新逻辑

```
用户请求 symbol + interval
  │
  ▼
查本地：SELECT MAX(close_time) WHERE symbol=? AND interval=?
  │
  ├── 本地无数据
  │     → fetch_klines(symbol, interval, limit=300)
  │     → 写入全部历史K线（排除最后一根，见下）
  │     → 返回全部数据
  │
  └── 本地有数据，last_close_time = T
        → 计算从 T 到现在差了多少根K线
        │
        ├── 差值 = 0（本地已是最新收盘K线）
        │     → 读本地最近 300 根
        │     → 从 Binance 补拉当前未收盘K线（1根）
        │     → 拼接返回
        │
        └── 差值 > 0
              → fetch_klines(symbol, interval, startTime=T)
              → 写入新增的已收盘K线（排除最后一根）
              → 读本地最近 300 根 + 当前未收盘K线
              → 返回
```
**关键规则：当前未收盘的最后一根K线永远不写库**，只实时返回。原因是未收盘K线的 OHLCV 是中间状态，写入后下次读到的是过期数据，破坏历史数据的不变性。

#### 文件位置

```
data/
  klines.db       ← SQLite 数据库文件
```

放在项目根目录的 `data/` 下，加入 `.gitignore`，不提交到版本库。

#### 对现有代码的影响

改动只在 `binance_service.py`，在 `fetch_klines` 前加一层缓存代理 `fetch_klines_cached`：

```
现在：fetch_klines() → 直接请求 Binance → 返回

改后：fetch_klines_cached() → 查本地 → 按需增量拉取 → 返回
                                              ↓
                                         fetch_klines()（只拉增量）
```

`app.py` 和所有算法层**零改动**，对上层完全透明。

### 6.3 用户手动标注落盘

#### 为什么必须落盘

用户手动画的笔/段/中枢是主观判断，与算法输出无关，无法通过重算还原。如果不持久化，用户调整好的标注在刷新页面后全部丢失。

#### 存储方案：JSON 文件

标注数据量极小（每张图几十个对象），结构灵活，JSON 文件直接可读可编辑，用户可以手动备份，也适合未来导入/导出功能扩展。

**文件命名规则**：

```
data/annotations/
  BTCUSDT_5m.json
  BTCUSDT_30m.json
  ETHUSDT_15m.json
  ...
```

每个 `symbol + interval` 组合对应一个文件，互不干扰。

**文件结构**：

```json
{
  "symbol": "BTCUSDT",
  "interval": "5m",
  "savedAt": 1715000000000,
  "turningPoints": [...],
  "segments": [
    {"fromIdx": 3, "toIdx": 11},
    ...
  ],
  "zhongshu": [
    {"fromIdx": 5, "toIdx": 9, "zg": 81200, "zd": 80500},
    ...
  ],
  "higherSegments": [...],
  "segmentZhongshu": [...],
  "deletedTurningPoints": [2, 7]
}
```
注意标注文件**只存索引和用户修改**，不存K线原始数据（K线在 SQLite 里），也不存自动计算的中间结果。

#### 自动保存触发时机

不需要用户手动点"保存"，以下操作自动触发保存：

- 手动画/删除一笔
- 手动画/删除一段或中枢
- 手动画高级别段或段中枢
- 点击"自动画段/中枢"等自动计算按钮后

实现方式：前端每次修改 `S.primary` 的标注数据后，调用 `POST /api/annotations/save`，后端写文件。防抖处理（500ms），避免频繁操作时重复写入。

#### 加载时自动恢复

用户加载 `BTCUSDT 5m` 时：

```
加载K线完成
  │
  ▼
GET /api/annotations/load?symbol=BTCUSDT&interval=5m
  │
  ├── 文件不存在 → 跳过，保持空白标注状态
  │
  └── 文件存在
        → 校验索引是否超出当前转折点范围
        → 有效标注恢复到 S.primary
        → 重新渲染图表
```

校验步骤很重要：如果 K线数据更新了（新增了更多历史数据），原来的 `fromIdx/toIdx` 可能指向不同的转折点，需要做范围检查，超界的标注丢弃或提示用户。

#### 新增后端接口

```
GET  /api/annotations/load?symbol=BTCUSDT&interval=5m   → 读取标注文件
POST /api/annotations/save                               → 写入标注文件（body 为标注JSON）
DELETE /api/annotations/clear?symbol=BTCUSDT&interval=5m → 清空标注文件
```

现有的 `/api/compute/export` 接口保留，作为手动导出完整数据（含K线）的功能，与自动保存标注是两个独立用途。

## 第七层：迁移步骤

整个改造分为**五个阶段**，每个阶段独立可测，不破坏现有功能。

## 遗留问题与已知限制

### 问题一：上下文周期的时间覆盖范围差异

采用新的配对方案后（倍数 4–8 倍），时间覆盖范围的差异已大幅收窄。例如 5m 取 300 根覆盖约 1 天，上下文周期 30m 取 300 根覆盖约 6 天，两者时间范围重叠部分充足，不会出现上下文结构完全在视口之外的问题。

极端情况（如 1m 主周期，上下文 5m 覆盖时长差异仍存在）的处理方式：上下文周期的 K 线取数与主周期时间范围做对齐，只渲染与当前视口有重叠的上下文结构，但计算仍用完整数据（保证结构完整性）。

### 问题二：手动编辑主周期笔/段后，上下文周期不自动更新

上下文周期完全独立计算，不受主周期手动编辑的影响。这在语义上是正确的（上下文周期是独立的时间粒度），但用户可能期望两者保持一致。

**建议处理**：在 UI 上明确标注"上下文周期为自动计算，不可手动编辑"，消除预期偏差。

### 问题三：部分交易对在大周期数据不足

某些新上线的交易对在日线或周线级别没有足够的历史数据，上下文周期计算会返回空结果。

**建议处理**：后端在上下文数据不足时返回空数组并附带 `contextWarning` 字段，前端优雅降级，只显示主周期数据。

### 问题四：app.js 的 2000 行单文件问题

当前 `app.js` 是全局状态 + 全局函数的单文件结构，状态重构后维护复杂度会进一步增加。

**建议处理**：阶段三完成后，将 `app.js` 拆分为：
- `state.js`：状态对象和 reset 函数
- `api.js`：所有 fetch 调用
- `chart.js`：SVG 渲染逻辑
- `ui.js`：侧边栏渲染、按钮交互

这不是功能改动，是维护性重构，可以在阶段四完成后单独进行。

## 平台最终形态：区间套交易决策系统

### 设计原则

用户坐在图表前真正的痛点不是"看不到买卖点"，而是：

> **"我现在不知道该不该动，不知道等的是什么，不知道条件满没满足。"**

所以平台的核心价值不是展示更多信息，而是**把复杂的多条件判断，降维成一个可执行的状态**。信息分三层，优先级从高到低：

```
第一层：现在该干什么（行动指引）   ← 平台重点呈现
第二层：为什么这么说（结构依据）   ← 辅助展示
第三层：原始数据（K线/笔/段）     ← 背景信息
```

大多数工具只做了第三层，用户自己做第一、二层，这是效率瓶颈所在。

### 背驰在状态机中的角色（P0）

#### 背驰的定义

背驰（Divergence）是缠论判断买卖点有效性的核心依据：

- **底背驰**：价格创新低，但 MACD 柱状面积（绝对值之和）小于前一段下跌的面积 → 下跌动能衰竭，买点有效性高
- **顶背驰**：价格创新高，但 MACD 柱状面积小于前一段上涨的面积 → 上涨动能衰竭，卖点有效性高

没有背驰的买卖点是"形态上的买卖点"，有背驰才是"有力支撑的买卖点"。

#### 背驰与状态机的关系

背驰不是状态机的**必要条件**，而是**置信度加权项**。设计如下：

```
无背驰的买卖点 → 状态机可以进入 CONFIRMING，但面板显示"低置信度"
有背驰的买卖点 → 状态机进入 CONFIRMING，面板显示"背驰确认，置信度高"
```

原因：如果把背驰设为必要条件，当 MACD 参数不理想或行情特殊时，状态机永远停在 WATCHING，反而让用户错过机会。更合理的做法是把背驰作为一个加权信号，让用户自己判断是否采纳。

#### 面板呈现方式

**有背驰时（CONFIRMING 状态）：**
```
┌─────────────────────────┐
│  ⚡ 30m 出现1买           │
│                         │
│  ✓ 4h 上升段             │
│  ✓ 30m 1买（80,320）      │
│  ✓ MACD 底背驰确认 ★     │  ← 加星标，置信度高
│  ○ 等待 5m 底分型         │
└─────────────────────────┘
```

**无背驰时（CONFIRMING 状态）：**
```
┌─────────────────────────┐
│  ⚡ 30m 出现1买（待确认）  │
│                         │
│  ✓ 4h 上升段             │
│  ✓ 30m 1买（80,320）      │
│  △ 暂无 MACD 背驰        │  ← 三角警示，置信度低
│  ○ 等待 5m 底分型         │
└─────────────────────────┘
```

#### 背驰计算必须迁移到后端

当前 MACD 背驰计算在前端 `app.js` 的 `computeMACD` / `detectDivergences` 里，存在两个问题：
1. 无法被后端状态机使用，`signal.py` 拿不到背驰结果
2. 没有单元测试，逻辑正确性无法保证

**迁移方案**：在阶段六（买卖点修复）时，将背驰检测逻辑移入 `algorithms.py`，作为 `compute_divergences(klines, turning_points)` 函数，后端计算后随买卖点数据一起返回给前端。前端只负责渲染，不再自己计算。

迁移后，`/api/compute/triple` 响应中的 `buySellPoints` 每条记录新增 `hasDivergence` 字段：
```json
{
  "idx": 14,
  "type": "buy",
  "label": "1",
  "hasDivergence": true,     ← 新增
  "macdArea": 125.3          ← 新增，当前段 MACD 面积，供前端展示
}
```

### 做空方向的对称设计（P1）

状态机在做多和做空方向完全对称，规则如下：

#### 做多（大周期上升段）

```
触发条件：大周期处于上升段，或刚出现段级别1买/2买
  ↓
WATCHING：等待操作周期出现 1买 / 2买 / 3买
  ↓
CONFIRMING：等待入场周期出现底分型（或小级别1买）
  ↓
READY：
  入场价 = 入场周期底分型第三根K线收盘价附近
  止损  = 入场周期底分型最低点下方
  目标一 = 操作周期中枢 ZG（上沿）
  目标二 = 操作周期当前段的预测终点（前高附近）
```

#### 做空（大周期下降段）

```
触发条件：大周期处于下降段，或刚出现段级别1卖/2卖
  ↓
WATCHING：等待操作周期出现 1卖 / 2卖 / 3卖
  ↓
CONFIRMING：等待入场周期出现顶分型（或小级别1卖）
  ↓
READY：
  入场价 = 入场周期顶分型第三根K线收盘价附近
  止损  = 入场周期顶分型最高点上方
  目标一 = 操作周期中枢 ZD（下沿）
  目标二 = 操作周期当前段的预测终点（前低附近）
```

#### 做空方向的失效条件（对称）

| 失效事件 | 退回状态 |
|---------|---------|
| 大周期下降段被反向段破坏 | → IDLE |
| 操作周期卖点被向上突破（价格高于1卖高点）| → WATCHING |
| 入场周期出现更高高点（顶分型入场结构破坏）| → CONFIRMING |

#### 方向判断逻辑

大周期方向由最新一段的方向决定：

```python
# signal.py 中的方向判断
context_segments = [...]   # 大周期段列表
if not context_segments:
    direction = "neutral"  # 无段，方向不明
else:
    last_seg = context_segments[-1]
    direction = "long" if last_seg["dir"] == "up" else "short"
```

特殊情况：大周期当前处于中枢内部震荡（无新段产生），方向为 `neutral`，状态机保持 IDLE，不触发任何方向的 WATCHING。
#### 面板方向标识

做多和做空在面板上用颜色和图标区分：

| 方向 | 颜色 | 图标 |
|------|------|------|
| 做多 | 绿色 `#51cf66` | ↑ |
| 做空 | 红色 `#ff6b6b` | ↓ |
| 震荡 | 灰色 `#888` | ↔ |

状态机同一时刻只跟踪一个方向（做多或做空），不会同时出现"多头信号"和"空头信号"。如果方向切换（大周期从上升段变为下降段），立即清空当前做多的所有状态，从 IDLE 重新开始跟踪做空方向。

### `signal.py` 的设计决策

#### 无状态 vs 有状态

`signal.py` 采用**无状态纯计算**设计：

```
每次调用传入：context数据 + primary数据 + entry数据
每次调用输出：当前 signal 状态（从数据中推导）
不持久化任何中间状态
```

原因：
- 缠论结构本身是完整的，从当前数据可以完整推导出"现在处于哪个状态"，不需要记住历史
- 无状态设计让函数可测试，输入确定则输出确定
- 避免持久化状态带来的一致性问题（例如数据更新后状态文件没有同步）

代价是每次轮询都做全量推导（约几十毫秒），对于 30 秒轮询间隔完全可接受。

#### 轮询频率与操作周期的关系

轮询频率不应固定为 30 秒，应随操作周期自动调整：

| 操作周期 | 轮询间隔 | 原因 |
|---------|---------|------|
| 1m–3m   | 10 秒   | K线变化快，需要及时感知 |
| 5m–15m  | 30 秒   | 平衡实时性与服务器压力 |
| 30m–1h  | 60 秒   | 结构变化慢，1分钟足够 |
| 4h–1d   | 300 秒  | 大周期无需频繁刷新 |

前端根据当前选择的操作周期自动设置轮询间隔，切换周期时重置轮询定时器。

---

### 平台不做的事

**平台只呈现结构和条件，不给出"应该买"的结论。**

正确的表达方式：
- "区间套条件已满足，当前结构与历史同类形态一致"
- "当前1买位置，止损在 X，若有效则最近目标在 Y"
- "风险收益比：1 : 2.1"

不应该出现的表达：
- "现在买入"
- "胜率 80%"（无统计依据）
- "强烈建议做多"

原因：买卖点是否有效仍依赖背驰判断（概率而非确定），用户的仓位和资金管理平台无法知道。给出确定性结论会导致用户盲目执行，出现亏损时归咎于工具。
